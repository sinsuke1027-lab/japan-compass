# テスト戦略

---

## 1. テスト方針

「壊れたら気づける」状態を維持することを目的とします。
完璧なカバレッジより、**重要な機能が動くことを保証する**テストを優先します。

---

## 2. テスト種別と責務

| 種別 | 対象 | ツール（例） | 実行タイミング |
|---|---|---|---|
| ユニットテスト | 関数・クラス単体 | Jest / pytest | コミット前・PR時 |
| 統合テスト | 複数モジュールの連携 | Jest / pytest | PR時 |
| E2Eテスト | ユーザー操作シナリオ | Playwright / Cypress | mainマージ前 |
| スモークテスト | デプロイ後の基本動作確認 | curl / カスタムスクリプト | デプロイ後 |

---

## 3. カバレッジ基準

| テスト種別 | 最低ライン | 目標 |
|---|---|---|
| ユニットテスト | 60% | 80% |
| 統合テスト | 主要APIエンドポイント全件 | - |
| E2Eテスト | クリティカルなユーザーフロー全件 | - |

---

## 4. 必ずテストすること（優先項目）

- [ ] 認証・認可（ログイン・ログアウト・権限チェック）
- [ ] データの作成・更新・削除（CRUD）
- [ ] バリデーション（正常値・異常値・境界値）
- [ ] エラーハンドリング（APIエラー・DB接続エラー等）
- [ ] 外部API連携（モック使用）

---

## 5. テストしないこと

- サードパーティライブラリの内部動作
- ビジュアルデザインの厳密な一致
- 環境依存の処理（→ モック化して対応）

---

## 6. テストファイルの配置規則

```
# ユニット・統合テスト
src/
├── components/
│   ├── Button.tsx
│   └── Button.test.tsx   ← 同階層に配置

# E2Eテスト
e2e/
└── auth.spec.ts
```

---

## 7. CI/CDでの自動テスト

- **PR作成時**: ユニットテスト・統合テストを自動実行
- **テスト失敗時**: GitHub Issueを自動起票（`test-failure-issue.yml`）
- **mainマージ後**: E2Eテスト・スモークテストを実行

---

## 8. Claude Code向け注意事項

- 新機能実装時は必ずテストも合わせて作成すること
- テストが落ちている状態でPRを作成しないこと
- モック・スタブは`__mocks__`または`tests/fixtures/`に配置すること
