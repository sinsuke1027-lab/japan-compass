# コミットメッセージ規約

## 形式（Conventional Commits 準拠）

```
{type}(#{Issue番号}): {概要}

{本文（任意）}

{フッター（任意）}
```

---

## type 一覧

| type | 用途 | 例 |
|---|---|---|
| `feat` | 新機能の追加 | `feat(#12): ユーザー認証機能を追加` |
| `fix` | バグ修正 | `fix(#34): ログイン時のnullエラーを修正` |
| `docs` | ドキュメントのみの変更 | `docs(#5): API仕様書を更新` |
| `style` | コードスタイルの変更（動作に影響なし） | `style: インデントを修正` |
| `refactor` | リファクタリング（機能追加・バグ修正なし） | `refactor(#78): 認証モジュールを整理` |
| `test` | テストの追加・修正 | `test(#12): 認証APIのユニットテストを追加` |
| `chore` | ビルド・設定ファイルの変更 | `chore: dependenciesを更新` |
| `ci` | CI/CD設定の変更 | `ci: AIレビューワークフローを追加` |
| `perf` | パフォーマンス改善 | `perf(#90): DB クエリを最適化` |
| `revert` | コミットの取り消し | `revert: feat(#12): ユーザー認証機能を追加` |

---

## 書き方のルール

1. **概要は50文字以内**
2. **日本語で記述**（プロジェクトに合わせて変更可）
3. **Issue番号は原則含める**（`chore`や`style`など自明なものは省略可）
4. **現在形・命令形で書く**（「追加した」ではなく「追加」）
5. 本文は**72文字で折り返す**
6. 本文と概要は**空行で区切る**

---

## 良い例・悪い例

```bash
# ✅ 良い例
feat(#12): ユーザーログイン機能を追加
fix(#34): パスワードリセットメールが送信されない問題を修正
docs(#5): README にセットアップ手順を追記

# ❌ 悪い例
update files          # typeなし・内容不明
fix bug               # 何のバグか不明
feat: いろいろ追加    # 範囲が広すぎる
WIP                   # 作業中のまま
```

---

## Claude Code向け注意事項

- 1コミット = 1つの変更（複数の変更を混ぜない）
- コミット前に`git diff`で変更内容を確認すること
- WIPコミットはPR前に`git rebase -i`でまとめること
