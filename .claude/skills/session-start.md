# スキル: セッション開始 — 状況分析と最善手の提案

## 使い方
```
/session-start
```

## 概要
MEMORY.md を読んで現在のプロジェクト状態を把握し、「今日最も価値のある行動」を
能動的に判断・提案する。ユーザーが何をすべきか迷わないよう、具体的なスキルと
理由までセットで提示するコーチ役スキル。

---

## 処理手順

### STEP 1: MEMORY.md を読んで状況を把握する

`MEMORY.md` を読む。

> **MEMORY.md が初期状態の場合（最終更新: YYYY-MM-DD のまま）:**
> 「MEMORY.md がまだ初期状態です。`/workflow-new-project` を実行してプロジェクトを開始してください。」と出力して終了する。

### STEP 2: 状況サマリーを出力する

```
## 📋 セッション開始

**フェーズ:** {現在のフェーズ}
**最終更新:** {日時}
**作業ブランチ:** {ブランチ名 or "-"}

### 前回完了したこと
{直近の完了タスク 上位3件}

### 積み残し
{未完了タスクの一覧。なければ「積み残しなし ✅」}

### 注意事項
{重要な決定事項・既知の問題から要点のみ。なければ省略}
```

### STEP 3: プロジェクト状態を多角的に分析する（ユーザーへの出力なし）

以下の観点でプロジェクト状態を内部評価し、最善手を決定する。

**① 緊急度チェック**
- 本番エラー・既知のバグ・ブロッカーがあれば最優先
- DB migration・デプロイ待ちなど「放置するとリスクになるもの」があれば次点

**② 流れの継続性**
- 前回セッションで中断した作業があれば「再開」が最も文脈コストが低い
- 直前に設計を完了した機能があれば「実装」に進むのが自然な流れ

**③ 準備の充足度**
- 実装したい機能はあるが `feature-design` が未完了 → 設計が先
- 設計は完了しているが specs/ が未更新 → 仕様書更新が先
- 3機能以上の設計が揃っている → 並列ビルドのタイミング
- specs は完了・実装も完了・PR が未作成 → PR 作成が先

**④ 技術的負債の蓄積度**
- 未適用の DB migration がある → 早期に適用を促す
- デプロイ待ちの Edge Function がある → 案内する
- テンプレートへのフィードバックが蓄積している → `/sync-template` を促す

### STEP 4: 最善手を能動的に提案する

分析結果に基づき、以下のフォーマットで **1つの最善手** を明確に提示する。

```
## 🎯 今日の最善手

**やること:** {具体的なアクション名}
**理由:** {なぜこれが今最も価値があるか・2〜3文で}
**使うスキル:** `/{skill-name}` または `{具体的なコマンド}`

---
### その後の流れ（参考）
1. {最善手の次にやること} → `/{skill}`
2. {さらにその次} → `/{skill}`
```

**最善手の選定ルール（優先順位）:**
1. 🔴 本番エラー・ブロッカーの修正
2. 🟠 DB migration・Edge Function デプロイなど放置リスクあり
3. 🟡 中断中の実装タスクの再開 → `/create-branch` + 実装
4. 🟡 設計完了済み機能が3件以上 → `/parallel-build`
5. 🟢 次の機能の設計 → `/feature-design`
6. 🟢 テンプレートへの反映 → `/sync-template`
7. 🔵 特になければ「次に取り組む機能のアイデア出し」を提案

**提案の姿勢:**
- 「〜してもよいですか？」ではなく「〜を進めることを推奨します」と断言する
- 複数の選択肢を羅列しない。最善手を1つ選んで理由を述べる
- ユーザーが別の作業を希望する場合は従う

---

## 出力例

```
## 📋 セッション開始

**フェーズ:** 実装中
**最終更新:** 2026-06-14
**作業ブランチ:** main

### 前回完了したこと
- Trip Planner / Memory / Stats & Badges を並列実装・マージ

### 積み残し
- DB migration（insights・activity_type）未適用
- Edge Function（fetch-url-title）未デプロイ

---

## 🎯 今日の最善手

**やること:** Supabase DB migration と Edge Function の適用
**理由:** PR #38〜#40 でマージした機能のうち、DB スキーマ変更と
Edge Function がまだ本番に反映されていません。これを放置すると
insights フィールドへの書き込みが失敗し続けます。
実装済み機能の動作確認も migration 適用後でないと正しく行えません。
**使うスキル:** Supabase ダッシュボード → SQL Editor で migration を実行、
その後 `supabase functions deploy fetch-url-title` を実行

---
### その後の流れ（参考）
1. Expo Go で動作確認 → 各画面の基本動作をチェック
2. 次の機能設計（ユーザーレビュー F-401）→ `/feature-design`
```

---

## 注意事項
- MEMORY.md を読むだけで、更新しない（更新は `/update-memory` の役割）
- セッション終了時は必ず `/update-memory` を実行するよう案内する
- ユーザーが最善手と異なる作業を希望したら、それに従う（強制しない）
