# /feature-design — 機能設計ヒアリング

機能追加・画面設計を始める前に、このスキルのフローで設計方針を合意する。
イメージのずれを防ぎ、手戻りを最小化するためのヒアリング手順。

## 実行フロー

### STEP 1: 全体像の提示

仕様書（specs/）を参照し、議論すべきポイントを表形式で一覧提示する。

```
| # | テーマ | 内容 |
|---|---|---|
| 1 | UI形式 | ... |
| 2 | コンテンツ範囲 | ... |
| 3 | オフライン戦略 | ... |
```

### STEP 2: 一つずつ確認

各ポイントを個別に取り上げ、以下の形式で提示してからユーザーの意見を求める。

- **選択肢**: A / B / C の比較表
- **レコメンド**: 推奨案とその理由（技術・UX・工数の観点から）
- **ユーザーの回答を待つ** — 先に進まない

### STEP 3: その他検討事項の確認

リストのポイントをすべて消化したら必ず問いかける：

> 「他に検討すべき点はありますか？」

ユーザーが新しい観点を提示した場合、STEP 2 の形式で同様に議論する。

### STEP 4: 最終結果の提示

全決定をまとめた表と画面構成イメージを提示し、確認を取る。

```
| # | テーマ | 決定 |
|---|---|---|
| 1 | UI形式 | 縦スクロールリスト |
...
```

ユーザーが「OK」「進めて」と言ったら実装フェーズへ移行する。

## 適用タイミング

- 新機能の実装開始前
- 既存画面の大幅な仕様変更時
- 複数の設計選択肢がある場合

## 適用不要なケース

- バグ修正
- 単純なコード変更・リファクタリング
- 文言・スタイルの微調整
