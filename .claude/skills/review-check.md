# スキル: 実装のセルフレビュー

## 使い方
```
/review-check
```

または特定ファイルを指定:
```
/review-check src/api/auth.ts
```

## 処理手順

1. `git status` を実行し、未コミットの変更がないことを確認する
   - 未コミットの変更がある場合はコミットまたはスタッシュしてから進む
2. 対象ファイル（または直近の変更差分）を取得する
3. 以下の観点でセルフレビューを実行する
4. テストを実行して全件パスすることを確認する
5. 問題があれば修正し、問題なければPR作成に進む

## チェック観点

### ✅ 仕様整合性
- `specs/`の関連仕様を参照し、実装が仕様に沿っているか確認
- 仕様にない機能を追加していないか確認
- `specs/requirements.md` が空テンプレートの場合はこの項目をスキップ

### ✅ セキュリティ
- `checklists/security-check.md`の項目を確認
- シークレットのハードコードがないか
- 入力バリデーションが実装されているか

### ✅ コード品質
- 命名が`docs/glossary.md`の用語に沿っているか
- 重複コードがないか（共通化できるか）
- エラーハンドリングが適切か
- デバッグ用の出力が残っていないか（`console.log` / `print` / `debugger` 等）

### ✅ テスト
- `docs/test-strategy.md`の基準を満たしているか
- 異常系のテストが存在するか
- テストを実際に実行して全件パスすることを確認する
  ```bash
  # プロジェクトに合わせていずれかを実行
  npm test        # Node.js / TypeScript
  pytest          # Python
  go test ./...   # Go
  ```
  テストが定義されていない場合はその旨を出力結果に記載する

### ✅ ドキュメント
- `docs/decision-log.md`に記録が必要な判断をしていないか
- コメントが適切に追加されているか

## 出力形式

```
## セルフレビュー結果

### 総評: ✅ 問題なし / ⚠️ 要修正 / ❌ ブロッカーあり

### 指摘事項
（問題がある場合のみ）

### 修正済み事項
（自動修正した内容）
```

## 注意事項

- このスキルはPR作成前に必ず実行する
- ブロッカーがある場合はPR作成を中止し、先に修正する
